Why Databases?

Imagine that you have a LOT of friends and you like to keep track of who they are, when you met them and when you're at the same events. So you decide to do a journal or diary and want it digital. Now you have a folder contain files, like:

  • PARTY 25-12-2024, containing details of the party and who attended (including any new friends)
  • John SMITH from school, containing details about John Smith (from School, not that other John Smith)
  • Friends List Update 25-12-2024, containing all the new frineds you made at the party.
  • Friends List Update 26-12-2024, containing all those you once called friends but are so unfriending.
  • Current friends list

As you can imagine, as the year go by, the number of files just keep on increasing and increasing. You start to drown in the digital weight of your journal. It takes AGES (unless you are lucky) to find out information. The only way to improve finding information would be to add MORE FILES to act as an index to your journal. The only way to make it easier to do anything with the journal would be to add other people to maintain or query the data. But adding people may mean it 'accidentally' gets deleted.

Or alternatively you can created your friends database. We'll call it FaceBase - totally unique name for a data BASE of friendly FACEs!

When starting to create a database we have to decide what information we want to store. For FaceBase we will store:

  • Different events and their attendees
  • Details about different friends (or former friends)
  • List of current friends and history of when people are added or unfriended

Please do not create your own FaceBase. Not because I've trademarked that awesome name or the idea itself. But rather, if you remember, we have rights to our data under Data Protection Act and GDPR. I don't think FaceBase will pass GDPR unless it's empty of any data relating to real people!

Now will we draft our first version of the data:
  1. EVENTS TABLE
    • Event ID
    • Name
    • Location
    • Start Date Time
    • End Date Time
    • Details of attendees
      • Name
      • Age
      • Is Friend?

  2. PEOPLE TABLE
    • Personal ID
    • Name
    • Gender
    • DoB
    • Is Friend Now?
    • Last Updated "Is Friend Now?"

  3. FRIENDS TABLE
    • Friend ID
    • Name
    • Last Added
    • Is Friend Now?
    • History of friendship
      • Updated At
      • Is Friend?
      • Reason for update

Now if we created this database we could organise the data much easier. Better yet we can create READ ONLY roles and let other people run queries. We can even setup backups so that if our data was lost (or someone corrupted it) then we could recover it. Therefore databases are awesome when:

  1. We have lots of data we want to keep updated (and we don't want a million files)
  2. We want to reduce keeping duplicate information
  3. We need to be able to get information (aka run queries) from this data
  4. We want to ensure consistency of data as database ensure it confirms to your schema
  5. Much better security as we can limit access using different user roles
  6. Also better security of data as we can regularly backup (and if needed restore)

Excel and other spreadsheets may appear similar to databases. But they are NOT databases. Using a spreadsheet as a database may result in data being lost or corrupted. More importantly the benefits above we don't get. So don't confuse Excel with databases.
